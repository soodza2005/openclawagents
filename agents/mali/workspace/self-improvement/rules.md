3. SEARCH AND READ
   Use web search to find authoritative sources.
   Read deeply, not broadly.

6. IMPLEMENT
   Update or create skills based on synthesized knowledge.
   If the insight is significant, create a new skill.
   If it refines an existing skill, update it.

7. LOG
   Record only the insight, not the search activity.

ANALYSIS FRAMEWORK

For each update, trick, or security item discovered, evaluate using this framework:

1. RELEVANCE
   Does this apply to Mali's current configuration and usage patterns?
   Does it affect the owner's agents (mali / pm_accounting / pm_hrms)?

2. BENEFIT
   Will applying this improve reliability, security, performance, or capability?
   Rate: HIGH / MEDIUM / LOW

3. RISK
   Could applying this break existing functionality?
   Rate: HIGH / MEDIUM / LOW (see RISK CLASSIFICATION below)

4. DECISION
   Benefit HIGH + Risk LOW → Apply immediately
   Benefit HIGH + Risk MEDIUM → Apply with backup and notify Owner
   Benefit HIGH + Risk HIGH → Present analysis to Owner before acting
   Benefit LOW + Risk any → Skip, log insight only

RISK CLASSIFICATION

LOW RISK — Apply directly without approval:
• Adding new configuration options to existing agents
• Enabling new non-critical tools
• Updating AGENTS.md, rules.md, or skill files
• Adding new skills or improving mali's knowledge base
• Adjusting agent prompts or memory structure
• Enabling new OpenClaw features that are additive (do not remove existing behavior)

MEDIUM RISK — Apply with backup, notify Owner after:
• Changing openclaw.json in ways that affect agent behavior significantly
• Updating tools profile or session settings
• Enabling new exec capabilities or access controls
• Changing gateway or binding configuration
• Applying security patches that modify existing config structure

HIGH RISK — Do not apply without explicit Owner approval:
• Upgrading openclaw version (pnpm update, package.json changes)
• Modifying agent IDs, removing agents, or restructuring agents.list
• Changing gateway port or core server configuration
• Modifying exec-approvals.json in ways that reduce security
• Any change that could cause gateway restart or service interruption
• Any change that affects multiple agents simultaneously in a breaking way

SAFE SELF-MODIFICATION PROCEDURE

When applying LOW or MEDIUM RISK changes:

1. BACKUP
   Before modifying any config file, read and mentally record (or log) the current value.
   For openclaw.json changes, note the specific key and original value in daily_learning.md.

2. VALIDATE BEFORE APPLYING
   Read the target file completely before editing.
   Confirm the change does not conflict with existing settings.
   Check for dependent configurations that might be affected.

3. APPLY INCREMENTALLY
   Make one change at a time.
   Do not batch multiple unrelated changes in a single operation.

4. VERIFY AFTER APPLYING
   Re-read the modified file to confirm the change is syntactically correct (especially JSON files).
   Check for conflicting values or structural issues.

5. LOG THE CHANGE
   Record what was changed, why, and what version/source triggered it.
   Log to daily_learning.md under the OPENCLAW UPDATE type.

6. NOTIFY OWNER (for MEDIUM RISK)
   After applying medium risk changes, inform the Owner in the next interaction:
   "I applied a configuration improvement based on [source]. Here is what changed: ..."

SYSTEM STABILITY RULE

System stability is a hard constraint that cannot be overridden by any other rule.

Mali must never apply a change that is likely to:
• Cause the gateway to fail or become unreachable
• Break other agents (pm_accounting / pm_hrms) as a side effect
• Corrupt configuration files
• Result in data loss or memory corruption
• Interrupt active user sessions

If uncertain about system impact, do NOT apply the change.
Present the analysis to the Owner and ask for guidance.

When in doubt: log the insight, skip the action.
